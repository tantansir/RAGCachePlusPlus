\begin{abstract}
Retrieval-Augmented Generation (RAG) increases serving latency because long retrieved contexts amplify prefill cost. Modern engines like vLLM provide Automatic Prefix Caching (APC), but RAG prompts with overlapping yet differently ordered documents break prefix alignment. We introduce RAGCache++, a prompt-layer scheduling policy that reorders documents to maximize prefix sharing using a knowledge tree and greedy ordering algorithm---requiring \textbf{zero engine modifications}. A TTFT-based cache-state feedback loop keeps the trie synchronized with vLLM's actual cache residency. On RTX 4060~Ti and RTX 4090, ordering reduces median TTFT by \textbf{20--33\%} (statistically significant), saving 66\% of prefill computation with $<$0.13\% overhead. The greedy algorithm achieves 97.5\% of oracle's TTFT reduction (prefix-length optimality ratio 0.997). Benefits persist under concurrent load (7--10\%), across GPU memory levels (29--31\%), generalize across architectures (Qwen2.5, Phi-3), and hold on a real TF-IDF-retrieved Wikipedia-style corpus (+26.8\% p50). Multi-hop QA evaluation confirms zero quality degradation ($\Delta$EM\,$=$\,0, $\Delta$F1\,$=$\,0).
\end{abstract}

\section{Introduction}
\label{sec:intro}
% ================================================================

Retrieval-Augmented Generation (RAG)~\citep{lewis2020rag} improves LLM factuality by injecting retrieved passages into the prompt, but this augmentation comes at a cost: prefill latency scales with the total number of input tokens, and RAG prompts are typically 2--10$\times$ longer than their non-augmented counterparts~\citep{gao2024ragsurvey}. For interactive applications where Time-To-First-Token (TTFT) directly impacts user experience, reducing prefill cost is critical.

Modern serving engines address this with \emph{prefix caching}: vLLM's Automatic Prefix Caching (APC)~\citep{vllm_apc} and SGLang's RadixAttention~\citep{zheng2024sglang} reuse KV cache blocks across requests when a new request shares an identical token-level prefix with a cached one. In vLLM, this mechanism is built on top of the block-based KV-cache design introduced by PagedAttention/vLLM~\citep{kwon2023pagedattention}.

The challenge for RAG is that prefix caching operates at the \emph{token} level, while RAG retrieval operates at the \emph{document} level. Two consecutive queries may retrieve four out of five identical documents, yet if those documents appear in different positions within their respective prompts, the token-level prefix diverges immediately, and no cache blocks are reused. This gap between document overlap and prefix overlap is the core inefficiency we target.

\textbf{Key insight.}  The observation that document ordering affects prefix cache reuse is well-established~\citep{jin2025ragcache,cachecraft2025,contextpilot2025}. Our contribution is not this observation but rather a precise, minimal implementation: because vLLM's APC matches prefixes by token-hash identity, we show that a simple prompt-layer reordering---requiring zero engine modifications---is sufficient to capture most of the available cache benefit.

\textbf{Our approach.} We propose RAGCache++, a lightweight prompt-level optimization that reorders documents to maximize prefix sharing with cached sequences. The system maintains a \emph{knowledge tree}---a trie indexed by document-ID sequences---that tracks which document orderings are currently in the KV cache. For each new request, a greedy algorithm walks the trie to find the longest cached prefix, places those documents first, and appends the remaining documents in retrieval-rank order. This approach:
\begin{itemize}
\item Requires \textbf{no modification} to vLLM or any serving engine---the core ordering policy (trie walk plus prompt build) is ${\sim}$150 LOC; the full integrated system including feedback loop and FastAPI middleware is ${\sim}$2{,}200 LOC (Section~\ref{sec:impl}).
\item Adds \textbf{negligible overhead}: the trie lookup is $O(k)$ where $k$ is the number of retrieved documents (typically 3--10).
\item Is not a new cache design but a \textbf{prompt-layer scheduling policy} that converts document overlap into APC-aligned token prefixes, complementing any prefix-caching engine.
\end{itemize}
While prior work (RAGCache~\citep{jin2025ragcache}, ContextPilot~\citep{contextpilot2025}) has explored cache-aware document scheduling, our contribution is to demonstrate that a \emph{minimal, engine-agnostic} implementation---with a near-optimal greedy algorithm (prefix-length optimality ratio 0.997) and online adaptivity---achieves competitive latency reductions (20--33\%) without the complexity of engine-level modifications. We additionally describe a retrieval-inference pipelining optimization as a secondary, orthogonal contribution.

\textbf{Research questions.} We organize the paper around five research questions:
\begin{itemize}
\item \textbf{RQ1.} Can pure prompt-layer scheduling (zero engine modification) capture most of the APC benefit? \emph{(Section~\ref{sec:eval}, Table~\ref{tab:main}; Section~\ref{sec:systems})}
\item \textbf{RQ2.} How close is $O(k)$ greedy to exhaustive oracle ordering? \emph{(Section~\ref{sec:design}: prefix-length ratio 0.997; Table~\ref{tab:baselines}: TTFT ratio 97.5\%)}
\item \textbf{RQ3.} Does the benefit generalize across real corpora, model architectures, and overlap regimes? \emph{(Section~\ref{sec:nq}; Section~\ref{sec:systems} Generality)}
\item \textbf{RQ4.} Can a TTFT-based feedback loop keep the trie synchronized with vLLM's actual cache state without engine instrumentation? \emph{(Section~\ref{sec:feedback}; Section~\ref{sec:trie_valid}, $r=0.97$; Section~\ref{sec:integrated}, 99.5\% accuracy)}
\item \textbf{RQ5.} Does reordering preserve answer quality, including on multi-hop reasoning? \emph{(EM $=$ 1.000 extractive; Table~\ref{tab:multihop}: $\Delta$EM $=$ $\Delta$F1 $=$ 0)}
\end{itemize}

Across these questions we evaluate on two GPU configurations, four model sizes spanning two architectures, and six workloads spanning a Jaccard overlap range of 0.06--0.88 (synthetic bursty, overlap sweep, geospatial, NQ-Open-derived, mixed-topic, and real-corpus TF-IDF), and show that simple prompt-layer ordering captures most of the available APC benefit.

\section{Background}
\label{sec:background}
% ================================================================

\subsection{RAG Serving Pipeline}

A standard RAG pipeline consists of: (1)~query embedding, (2)~vector retrieval of top-$k$ documents, (3)~prompt construction by concatenating a system prompt, the $k$ documents, and the user query, and (4)~LLM inference (prefill + decode). Steps~(1)--(2) typically take 10--50\,ms with an optimized index, while step~(3) is negligible. Step~(4) dominates latency: prefill cost scales as $O(n \cdot d)$ where $n$ is the prompt length and $d$ is the model dimension.

\subsection{vLLM Automatic Prefix Caching}

vLLM manages KV cache memory in fixed-size \emph{blocks} via PagedAttention~\citep{kwon2023pagedattention}. On top of this block-based KV-cache design, vLLM provides Automatic Prefix Caching (APC), which reuses cached KV blocks when a new request shares the same token-level prefix as a previously processed request~\citep{vllm_apc}.

Concretely, APC identifies reusable prefix blocks using block-level hashing over the prefix state; once a prefix mismatch occurs, subsequent blocks can no longer be reused for that request, so the remaining prompt must be prefetched again~\citep{vllm_apc}.

\textbf{Implication for RAG.} Because the hash is \emph{chained}, even a single-token difference at position $j$ invalidates all blocks at positions $\geq j$. This means that if two RAG prompts share documents $\{A, B, C\}$ but arrange them as $[A, B, C, D, E]$ vs.\ $[B, A, C, D, F]$, the prefix diverges at the very first document and no KV blocks are reused---despite 60\% document overlap.

\subsection{Related Work}

The observation that document ordering affects prefix cache reuse in RAG systems is well-established. Several concurrent and prior systems address this from different angles; we position our work within this landscape.

\textbf{RAGCache}~\citep{jin2025ragcache} is the closest prior work. It organizes cached KV states in a knowledge tree indexed by document sequences, uses a PGDSF replacement policy with multi-tier placement, and includes cache-aware \emph{request-level} scheduling and dynamic speculative pipelining. RAGCache modifies the serving engine internals. Our work shares the knowledge-tree concept but differs in scope: we perform \emph{intra-prompt} document reordering at the prompt-construction layer without any engine modification. Table~\ref{tab:baselines} shows that our greedy trie captures 97.5\% of oracle TTFT reduction in a single-run comparison against Sorted, Recency, Retrieval, and exhaustive-oracle baselines; because RAGCache's cache-aware scheduler is itself designed to approximate oracle ordering, this suggests that the scheduling abstraction---whether realized as RAGCache's engine-level scheduler or our prompt-layer ordering---captures most of the available benefit. A direct head-to-head comparison with RAGCache on the same stack would require re-implementing its engine-level modifications and is left to future work.

\textbf{ContextPilot}~\citep{contextpilot2025} is a concurrent system that also reorders context blocks to maximize KV-cache reuse, using a context index and de-duplication techniques. Like our work, it operates at the prompt layer with negligible overhead (${\sim}$0.047\,ms). Our approach differs in using a trie-based greedy algorithm with a measured prefix-length optimality ratio of 0.997, online adaptivity under distribution shift, and comprehensive evaluation across architectures and workloads. The convergence of these independent efforts validates the importance of cache-aware ordering as a systems abstraction.

\textbf{Cache-Craft}~\citep{cachecraft2025} reports that in production RAG workloads, exact prefix caching applies to only ${\sim}$8\% of requests because retrieved chunks and their ordering vary across queries. Their solution identifies reusable chunk-caches and selectively recomputes the rest. This production finding directly motivates our work: we aim to \emph{increase} the fraction of prefix-cacheable requests by aligning document ordering with cached prefixes.

\textbf{TurboRAG}~\citep{turborag2024} goes beyond strict ordering constraints by pre-computing chunk-level KV caches offline and stitching them at inference via reordered-RoPE, enabling reuse even when document order differs. This represents a complementary direction: TurboRAG relaxes the prefix constraint at the attention level, while we maximize reuse \emph{within} the prefix constraint via ordering.

\textbf{CacheBlend}~\citep{yao2025cacheblend} enables non-prefix KV reuse by selectively recomputing ${\sim}$15\% of tokens per layer. Like TurboRAG, it addresses the harder problem of position-independent reuse; our approach converts the problem to pure prefix reuse via ordering, which is simpler but sufficient when document overlap is moderate.

\textbf{CacheGen}~\citep{lmcache2024} compresses KV caches for efficient storage and streaming. This is orthogonal to ordering optimization.

\section{Methodology}
\label{sec:design}
% ================================================================

\subsection{Overview}

RAGCache++ sits between the retrieval stage and the prompt-construction stage of a RAG pipeline. Given a set of retrieved document IDs, it reorders them to maximize the token-level prefix shared with recently served requests, then constructs the prompt with documents in the optimized order. The serving engine (vLLM with APC enabled) handles KV cache management transparently.

\subsection{Knowledge Tree}

The knowledge tree is a trie where each path from root to leaf represents a document ordering that has been recently served. Each node stores a document ID and metadata (access count, timestamp).

\begin{itemize}
\item \textbf{Insert}: After each request is served, the ordered document sequence is inserted into the trie. Existing paths are updated (access count incremented); new branches are created as needed. Complexity: $O(k)$ per request.
\item \textbf{Prefix match}: Given a candidate document set, traverse the trie to find the longest path where every document along the path belongs to the candidate set and has live (cached) KV metadata. Complexity: $O(k)$ per request.
\end{itemize}

\subsection{Greedy Ordering Algorithm}

Algorithm~\ref{alg:ordering} describes the ordering procedure. Starting at the trie root, we greedily extend the cached prefix by choosing, at each level, a child node whose document ID is in the remaining candidate set. When no cached continuation exists, remaining documents are appended in their original retrieval-rank order to preserve relevance ranking.

\begin{algorithm}[t]
\caption{Cache-aware document ordering}
\label{alg:ordering}
\begin{algorithmic}[1]
\REQUIRE Retrieved doc IDs $D = \{d_1, \ldots, d_k\}$ (in retrieval rank), knowledge tree $T$
\ENSURE Ordered sequence $O$
\STATE $O \leftarrow []$, $R \leftarrow D$, $\textit{node} \leftarrow T.\text{root}$
\WHILE{$R \neq \emptyset$}
    \STATE $\textit{found} \leftarrow \texttt{false}$
    \FOR{each $d \in R$}
        \STATE $c \leftarrow \textit{node}.\text{getChild}(d)$
        \IF{$c \neq \texttt{null}$ \AND $c.\text{isCached}()$}
            \STATE Append $d$ to $O$; remove $d$ from $R$
            \STATE $\textit{node} \leftarrow c$; $\textit{found} \leftarrow \texttt{true}$; \textbf{break}
        \ENDIF
    \ENDFOR
    \IF{$\textit{found} = \texttt{false}$}
        \STATE Append remaining $R$ in retrieval-rank order to $O$
        \STATE \textbf{break}
    \ENDIF
\ENDWHILE
\STATE \textbf{return} $O$
\end{algorithmic}
\end{algorithm}

\textbf{Objective.} The ordering problem can be stated as: given a candidate set $D$ and a trie $T$ recording recently cached document sequences, find the permutation $\pi$ of $D$ that maximizes the number of leading documents matching a path in $T$ (i.e., the cached prefix length). Because APC uses chained hashes, each additional matched document avoids prefilling all of that document's tokens, so maximizing prefix length directly minimizes prefill cost. The greedy trie walk is a low-overhead heuristic---$O(k)$ per request---chosen for practical simplicity; it is optimal when the trie has at most one matching child per level (often the case in our synthetic bursty setting), but does not perform multi-step lookahead.

\textbf{Approximation quality.} To quantify how close the greedy heuristic is to optimal, we compute the ratio of greedy prefix length to oracle prefix length (exhaustive $k!$ search) for each query in a 200-query trace ($k{=}5$, overlap\,$=$\,0.6). The greedy algorithm achieves a mean prefix-length optimality ratio of \textbf{0.997} (1.0\,$=$\,perfect), solving 98.5\% of queries optimally. The minimum ratio observed is 0.80 (4 out of 5 documents matched vs.\ 5 out of 5), occurring only for queries at region boundaries. Note that this is the ratio of matched prefix \emph{lengths}; the ratio of TTFT \emph{reductions} is separately reported as 97.5\% in Section~\ref{sec:systems} (Table~\ref{tab:baselines}).

\textbf{Fallback behavior.} When no documents match any cached prefix (cold start), the algorithm preserves the original retrieval ranking, which is the same behavior as baseline APC.

\subsection{Knowledge Tree as an Approximate Cache Index}

The knowledge tree tracks which document sequences have been \emph{served}, not which KV blocks are \emph{resident} in vLLM's cache. Because vLLM manages eviction internally (LRU-based block eviction under memory pressure), the trie may contain stale entries for sequences whose KV blocks have been evicted. In practice, this mismatch is benign under sequential evaluation with sufficient GPU memory. Under concurrent load or memory pressure, we address stale predictions with the TTFT-based feedback loop described in Section~\ref{sec:feedback}, which detects and prunes stale trie entries automatically.

\subsection{Integration with vLLM}

RAGCache++ does not modify vLLM internals. The integration consists of three components:
\begin{enumerate}
\item A \textbf{prompt builder} that accepts retrieved document IDs and contents, calls the ordering algorithm, and constructs the final prompt string.
\item The \textbf{knowledge tree} maintained in the application layer, updated after each request.
\item Standard vLLM \texttt{LLM.generate()} calls with \texttt{enable\_prefix\_caching=True}.
\end{enumerate}
The core ordering policy (prompt builder + trie walk) adds approximately 150 LOC on top of a standard RAG pipeline; the full integrated system including the feedback loop, cache manager, and FastAPI middleware is ${\sim}$2{,}200 LOC (detailed in Section~\ref{sec:impl}).

\subsection{Cache-State Feedback Loop}
\label{sec:feedback}

A key limitation of the trie-as-cache-proxy design is that the knowledge tree tracks which document sequences have been \emph{served}, not which KV blocks are \emph{resident} in vLLM's cache. Under memory pressure, vLLM may evict blocks that the trie still considers cached, leading to stale predictions.

We close this gap with a \emph{TTFT-based cache-state feedback loop}. The system maintains a running estimate of cold-start TTFT (via exponential moving average over cache-miss observations). After each request, it compares the trie's predicted reuse fraction ($\hat{r} = \text{prefix\_len} / k$) with the TTFT-inferred actual reuse ($r = 1 - T_{\text{actual}} / T_{\text{cold}}$). When the trie predicts a significant prefix hit ($\hat{r} > 0.3$) but the observed TTFT is near-cold ($r < 0.15$), the system detects a \emph{mismatch}---indicating that vLLM has evicted the expected blocks---and marks the predicted path as stale (\texttt{tier="none"}).

This feedback mechanism integrates all system components into a closed-loop pipeline:
\begin{enumerate}
\item \texttt{CacheManager} tracks multi-tier cache state with PGDSF eviction priorities.
\item \texttt{KnowledgeTree} provides prefix-aware ordering via greedy trie walk.
\item \texttt{CacheStateFeedback} detects trie-vs-reality mismatches from TTFT observations.
\item Stale trie paths are pruned automatically, keeping the trie synchronized with vLLM's actual cache residency.
\end{enumerate}

The entire integrated pipeline (\texttt{VLLMCacheProxy}) adds only \textbf{0.049\,ms} per request ($<$0.1\% of TTFT), achieves 99.5\% prediction accuracy, and requires zero vLLM modifications.

\subsection{Retrieval-Inference Pipelining}

\noindent\textit{Note: Retrieval-inference pipelining is a secondary, orthogonal optimization that benefits only cold-start and cache-eviction-recovery requests. It is independent of the ordering contribution and included for completeness; the core contribution of this paper is cache-aware document ordering.}

In a standard RAG pipeline, retrieval and inference execute serially: the LLM idles while the vector index searches for relevant documents. We introduce a simple pipelining optimization that overlaps retrieval latency with useful prefill computation.

The key observation is that the \emph{system prompt} (instruction header) is identical across all requests. When APC is enabled, the system prompt's KV blocks are cached after the first request. We exploit this by issuing a lightweight warmup call (system prompt only) concurrently with the retrieval query. By the time retrieval returns with document IDs, the system prompt is already cache-resident, and the full prompt's prefill need only compute the document and query tokens.

Concretely, let $T_{\text{ret}}$ be retrieval latency, $T_{\text{sys}}$ the system-prompt prefill time, $T_{\text{warm}}$ the TTFT once the system prompt is cached, and $T_{\text{cold}}$ the cold-start TTFT. Empirically $T_{\text{cold}} = T_{\text{sys}} + T_{\text{warm}} + \delta$, where $\delta$ is a small engine-setup overhead that the warmup call amortizes. Without pipelining, end-to-end TTFT is $T_{\text{ret}} + T_{\text{cold}}$. With pipelining, the warmup call runs concurrently with retrieval, giving
\begin{equation*}
\text{pipelined TTFT} \;=\; \max(T_{\text{ret}}, T_{\text{sys}}) + T_{\text{warm}},
\end{equation*}
and per-request saving
\begin{equation*}
T_{\text{ret}} + T_{\text{cold}} - \max(T_{\text{ret}}, T_{\text{sys}}) - T_{\text{warm}} \;=\; \min(T_{\text{ret}}, T_{\text{sys}}) + \delta.
\end{equation*}
Thus the saving grows with $T_{\text{ret}}$ up to $T_{\text{sys}}$ (at which point it is capped at $T_{\text{sys}} + \delta = T_{\text{cold}} - T_{\text{warm}}$, the full overlapped prefill plus engine setup). After the first request the system prompt is cache-resident, so the benefit is limited to cold starts and cache eviction events. Implementation requires only a \texttt{concurrent.futures.ThreadPoolExecutor} to dispatch the warmup call alongside retrieval---no engine modification is needed.

\subsection{Experimental Setup}

\textbf{Hardware.} We evaluate on two independent configurations (see Appendix~\ref{app:config} for details): (1)~NVIDIA RTX 4060~Ti (8\,GB VRAM) with Qwen2.5-1.5B-Instruct~\citep{qwen25tr} on vLLM~0.8.5, and (2)~NVIDIA RTX 4090 (24\,GB VRAM) with Qwen2.5-7B-Instruct~\citep{qwen25tr} on vLLM~0.17.1.

\textbf{Workload.} We generate a synthetic corpus of documents (${\sim}200$ tokens each, standardized length so that each document spans a similar number of KV cache blocks, approximating the token-prefix depth per chunk) grouped into regions of 10 documents each (100 docs / 10 regions on 4060~Ti; 500 docs / 50 regions on 4090). Queries arrive in bursts of 10 from the same region with 60\% document overlap between consecutive queries (Jaccard~$\approx 0.52$). This controlled setup models bursty RAG patterns where sequential queries share retrieved context (e.g., geographic locality, topic continuity). We acknowledge that production RAG workloads may exhibit different overlap distributions.

\textbf{Strategies compared.} We evaluate four strategies, each with a \emph{fresh} LLM instance to prevent warm-cache confounding:
\begin{itemize}
\item \textbf{No Cache}: APC disabled. Full prefill for every request.
\item \textbf{APC + Retrieval}: APC enabled, documents in retrieval-rank order.
\item \textbf{APC + Sorted}: APC enabled, documents sorted lexicographically by ID.
\item \textbf{APC + Optimized (Ours)}: APC enabled, documents ordered by the knowledge tree algorithm.
\end{itemize}

\textbf{Metrics.} We measure TTFT (Time-To-First-Token) by generating only 1 output token per request so that measured latency is dominated by prefill. Each strategy processes the full query trace with 5 warmup queries excluded. We report p50, p95, and mean TTFT. We also report an approximate \emph{effective cache-hit proxy} (Fast\%$^\dagger$): the fraction of post-warmup requests whose TTFT falls below $0.6\times$ the median warmup TTFT. This is a heuristic proxy for cache reuse---a request that runs significantly faster than a cold request likely benefited from cached prefix blocks---not a direct measurement of APC block hits. The more informative metric is TTFT itself, since it captures both hit \emph{frequency} and hit \emph{depth} (how many prefix blocks were reused).

\textbf{Cross-table baselines.} Because each table in Section~\ref{sec:eval} comes from an independent benchmark run with a freshly loaded LLM instance and its own query trace (necessary to prevent warm-cache confounding), the No-Cache p50 baseline varies across tables on the same 4090 hardware (e.g., 121.3\,ms in Table~\ref{tab:main}, 121.5\,ms in Table~\ref{tab:baselines}, 114.7\,ms at batch\,=\,1 in Table~\ref{tab:concurrent}, 110.3\,ms in Table~\ref{tab:spatial}). Within each table, percentage improvements are always computed against that table's own APC+Retrieval baseline, so comparisons inside a single table are internally consistent; absolute cross-table comparisons should be read with this run-to-run variance in mind.

\begin{enumerate}
\item Sorted ordering distributes prefix matches across many short prefixes (e.g., matching only the first 1--2 documents before diverging).
\item Optimized ordering concentrates matches into fewer but longer prefixes (e.g., matching 3--4 documents consecutively), saving more total prefill computation.
\end{enumerate}
This demonstrates that \emph{prefix length}, not just hit count, is the relevant metric for cache efficiency in prefix-caching systems.

\begin{itemize}
\item The left-most point (labelled ``0.0'' on the $x$-axis) is \emph{not} true zero overlap: the trace generator enforces $\max(1, \lfloor k \cdot \text{overlap} \rfloor) = 1$ shared document even at overlap~$=0$, which combined with region-burst sampling yields an \emph{actual} Jaccard of 0.39. \item The \textbf{peak improvement of 30.0\%} occurs at overlap fraction 0.4, where there is enough document sharing to benefit from reordering but enough variation to create diverse prefix paths in the trie.
\item At \textbf{very high overlap (0.8)}, the improvement is minimal because queries share nearly all documents and retrieval-rank order already produces near-maximal prefix overlap.
\end{itemize}

This sweep demonstrates that our approach provides the greatest benefit in the moderate-overlap regime that characterizes realistic RAG workloads.

\subsection{Answer Quality Preservation}

We evaluate whether document reordering degrades answer quality using Qwen2.5-7B-Instruct.
On 97 extractive QA questions (Qwen2.5-7B, answer explicitly embedded in one of 5 passages, strict ``output ONLY the answer'' prompt, \texttt{max\_tokens}$=$16; 3 cases filtered due to passage-length overflow), both orderings achieve \textbf{EM\,$=$\,1.000} with $\Delta$EM\,$=$\,0. This confirms that when the model is capable of extractive QA, reordering does not change the extracted answer.

\textbf{Interpretation.} Document reordering preserves answer quality because it changes only the \emph{order} of retrieved passages within the prompt, not their content. For extractive QA tasks where passages act as interchangeable evidence, the ordering is immaterial to the model's ability to locate and extract the answer.

\subsection{Workload Diversity}

We evaluate three workloads at different overlap regimes to show that the main-benchmark result (Section~\ref{sec:eval}, synthetic Jaccard\,$\approx$\,0.52) is not a single data point. Each of the three uses an independent trace and its own No-Cache baseline; we report within-workload percentages only, consistent with the cross-run note in Section~\ref{sec:eval}.

\paragraph{NQ-Open (high overlap, Jaccard = 0.88).}
\label{sec:nq_open}
To evaluate RAGCache++ on a workload derived from real questions, we construct a corpus from the NQ-Open dataset\footnote{NQ-Open is the open-domain variant of Google's Natural Questions benchmark~\citep{kwiatkowski2019nq}: each item is a real Google search query paired with short answer strings extracted from English Wikipedia. HuggingFace mirror: \url{https://huggingface.co/datasets/google-research-datasets/nq_open}.} using 500 question-answer pairs. We generate topically-clustered Wikipedia-style passages grouped by answer similarity and retrieve top-5 passages per query using TF-IDF similarity. This produces a \emph{high-overlap} trace (mean Jaccard $=0.88$). At this operating point APC alone already achieves 63\% TTFT reduction over the same run's No-Cache baseline, and ordering adds only a further 2.3\%---retrieval order is already near-optimal for prefix alignment, matching the overlap-sensitivity curve in Figure~\ref{fig:overlap}.

\paragraph{Mixed-topic (moderate overlap, Jaccard = 0.31).}
To test a workload with heterogeneous, moderate overlap, we generate traces where each query has a 30\% probability of switching to a random topic region, producing a mix of within-burst and cross-topic queries (mean Jaccard\,$=$\,0.31, p50\,$=$\,0.43). Optimized ordering reduces p50 TTFT from 94.3\,ms to 79.0\,ms (\textbf{16.3\% improvement}). This is the regime where ordering gains scale most naturally: multi-topic RAG deployments where users shift between related subjects (Jaccard\,$\approx$\,0.3--0.5).

\paragraph{Real-corpus TF-IDF (low natural overlap, Jaccard = 0.06).}
\label{sec:real_corpus}
To address the concern that all workloads above still rely on \emph{some} form of topic clustering, we additionally construct a diverse corpus of 1{,}000 Wikipedia-style passages spanning 20 topics (machine learning, astronomy, medicine, history, geography, physics, biology, economics, technology, literature, etc.) and use TF-IDF retrieval to produce a query trace with \emph{naturally arising} document overlap---no controlled overlap parameter, no topic clustering. The resulting mean Jaccard is only $0.06$, characteristic of unconstrained diverse-topic retrieval. Table~\ref{tab:real_corpus} (values in this table correspond only to this real-corpus experiment) shows that even under this very low natural overlap, optimized ordering reduces p50 TTFT by \textbf{26.8\%} (24.8\,ms vs.\ 33.9\,ms) and mean TTFT by 20.4\%, confirming that the benefit is not an artifact of high controlled overlap---a very small number of short prefix hits still translate into real TTFT savings.

\subsection{System Validation}
\label{sec:trie_valid}

Section~\ref{sec:design} notes that the knowledge tree approximates cache residency rather than querying actual vLLM APC state. To validate this approximation, we measure the correlation between the trie's predicted reuse fraction (predicted prefix length / $k$) and the TTFT-inferred actual reuse fraction ($1 - T_{\text{actual}} / T_{\text{cold}}$). Under normal conditions ($\texttt{gpu\_mem} = 0.90$), Pearson $r = 0.972$ and Spearman $\rho = 0.925$; under memory pressure ($\texttt{gpu\_mem} = 0.78$), Pearson $r = 0.965$ and Spearman $\rho = 0.925$. The small degradation under pressure confirms that the trie remains an accurate cache proxy even with a smaller KV cache pool.


\paragraph{Integrated pipeline.}
\label{sec:integrated}

We evaluate the full integrated pipeline (\texttt{VLLMCacheProxy}) against bare vLLM and the simple ordering wrapper.


The integrated proxy achieves the same latency reduction as the simple wrapper (29.7\% p50 improvement over bare vLLM) while adding cache-state synchronization via the feedback loop. Per-component profiling shows the system overhead is dominated by prompt construction (0.032\,ms) and trie ordering (0.017\,ms); inference accounts for 99.9\% of wall-clock time.

Under eviction pressure (\texttt{gpu\_mem}\,=\,0.78), the feedback loop achieves \textbf{99.5\% prediction accuracy}---only 1 mismatch out of 200 queries, triggering 1 trie pruning event. The sequential workload produces few evictions even at reduced memory, so the accuracy mainly reflects that the trie is an accurate cache proxy in this regime; the feedback mechanism correctly detects and prunes the rare stale entries that do occur.

\subsection{Key Code Snippets}

\paragraph{Greedy trie-based ordering.} The 25-line realization of Algorithm~\ref{alg:ordering} is the entire ``policy'' that yields the 20--33\% TTFT reduction. It walks the knowledge tree and only follows children whose KV is currently resident (\texttt{tier != "none"}); on the first miss, the remaining documents fall back to retrieval rank order.

\begin{lstlisting}[caption={Greedy ordering (\texttt{prompt\_builder.py}). The cold-path loop appends EVERY remaining document (matching Algorithm~\ref{alg:ordering}), not just one.},label={lst:order}]
def optimize_doc_order(doc_ids, kt):
    ordered, remaining = [], set(doc_ids)
    node = kt.root
    while remaining:
        found = False
        # Hot path: extend the cached prefix by one cached child
        for d in list(remaining):
            child = node.get_child(d)
            if (child is not None
                and child.kv_metadata is not None
                and child.kv_metadata.tier != "none"):
                ordered.append(d); remaining.discard(d)
                node = child; found = True
                break   # only break the inner for; outer while continues
        if not found:
            # Cold path: append ALL remaining docs in retrieval-rank
            # order, then exit the outer while loop.
            for d in doc_ids:
                if d in remaining:
                    ordered.append(d)
            break
    return ordered
\end{lstlisting}

\paragraph{TTFT-based mismatch detection.} Without engine instrumentation, the proxy infers vLLM's actual cache state by comparing the trie's predicted reuse fraction with TTFT-implied reuse. A predicted hit ($\hat{r}>0.3$) that lands near cold-start ($r<0.15$) is flagged for trie pruning. The cold-TTFT estimate is updated online via EMA over observed misses.

\begin{lstlisting}[caption={Cache-state feedback (\texttt{serving.py}).},label={lst:feedback}]
def check_mismatch(self, predicted_prefix_len, k, ttft_ms):
    self.total_observations += 1
    pred_reuse   = predicted_prefix_len / max(k, 1)
    actual_reuse = max(0.0, 1.0 - ttft_ms / max(self.cold_ttft, 1.0))
    self.history.append((pred_reuse, actual_reuse))
    if pred_reuse > 0.3 and actual_reuse < 0.15:
        self.mismatches += 1
        return True              # trie thinks hit, vLLM evicted
    return False
\end{lstlisting}

\paragraph{Integrated serve loop.} The proxy threads ordering, prompt build, vLLM call, feedback, pruning, and admission into a single call with seven explicit stages, yielding the per-component overhead numbers reported in Table~\ref{tab:profiling}.

\begin{lstlisting}[caption={\texttt{VLLMCacheProxy.serve\_request} (abridged).},label={lst:serve}]
def serve_request(self, query, doc_ids, doc_contents, max_tokens=1):
    # 1. Greedy trie walk -> ordered ids + predicted prefix length
    ordered, predicted = self._greedy_walk(doc_ids)
    # 2. Build prompt
    prompt, _ = build_rag_prompt(query, ordered, doc_contents)
    # 3. vLLM inference (APC enabled at LLM construction)
    t0 = time.perf_counter()
    out = self.llm.generate([prompt], sp)
    ttft_ms = (time.perf_counter() - t0) * 1000
    # 4. Feedback loop
    self.feedback.update_cold_estimate(ttft_ms, predicted, len(doc_ids))
    miss = self.feedback.check_mismatch(predicted, len(doc_ids), ttft_ms)
    # 5. Prune stale paths if vLLM evicted them
    if miss:
        self._prune_path(ordered[:predicted])
    # 6. Admit into multi-tier cache state
    metas = [self.cache_manager.admit(d, 200, retrieval_rank=i+1)
             for i, d in enumerate(ordered)]
    self.cache_manager.insert_sequence(ordered, metas)
    self.cache_manager.step()
    return {"ttft_ms": ttft_ms, "predicted_prefix_len": predicted, ...}
\end{lstlisting}

The proxy is also wrapped as a FastAPI service (\texttt{RAGCacheMiddleware}) so an existing RAG application can switch to cache-aware ordering by replacing the \texttt{LLM} object with one HTTP endpoint, without touching its serving stack.

\subsection{Engineering Details and Trade-offs}

\begin{itemize}
\item \textbf{vLLM API surface.} We pin to vLLM~$\geq$~0.8.0 for stable APC behavior; the proxy invokes only the public \texttt{LLM.generate} interface with \texttt{enable\_prefix\_caching=True} and standard \texttt{SamplingParams}. No internal vLLM symbols are imported, which makes the integration robust to engine-version drift.
\item \textbf{Trie memory footprint.} Nodes use \texttt{\_\_slots\_\_} to bound per-node memory at ${\sim}120$\,B. In our 200-query benchmark the trie holds $<$1\,MB, well below any meaningful host budget.
\item \textbf{Reproducible eviction.} \texttt{CacheManager.step()} advances a logical timestamp consumed by the PGDSF priority, decoupling eviction order from wall-clock noise. All benchmark scripts accept a \texttt{seed} argument; the multi-seed statistics in Section~\ref{sec:eval} are produced by re-running with seeds $\{0,1,2\}$ and aggregating with a $t$-distribution 95\% CI.
\item \textbf{Fallback path.} When the trie is empty (cold start) or no document matches a cached child, the algorithm degenerates to retrieval-rank order. \emph{Algorithmically}, this makes the proxy no worse than bare APC in the fallback case; empirical TTFT can still differ by a few milliseconds of measurement noise across independent runs.
\end{itemize}

\section{Discussion}
\label{sec:discussion}
% ================================================================

\textbf{Prompt-layer cache scheduling.}
RAGCache++ introduces \emph{prompt-layer cache scheduling} for APC-based RAG serving, analogous to how disk I/O schedulers reorder requests to exploit spatial locality without modifying the disk firmware. The ordering policy does not modify the cache mechanism itself (APC) but determines which cache lines are populated and in what order, directly controlling the prefill--reuse trade-off. Our profiling confirms this is a genuine systems effect: optimized ordering saves 66\% of prefill computation at p50 (Table~\ref{tab:cache_eff}), reduces inference latency by 29\% (Table~\ref{tab:profiling}), with zero throughput regression and zero memory overhead. The key architectural insight is that a \emph{thin scheduling layer} between retrieval and serving can recover substantial cache locality that the retrieval stage destroys---without requiring engine-level changes. This makes the policy composable with any APC-enabled system and deployable as a drop-in middleware component.
We emphasize that the contribution is \emph{cache-aware document ordering as a scheduling abstraction}. The trie-based greedy algorithm achieves 97.5\% of oracle TTFT reduction with $O(k)$ cost (prefix-length optimality ratio 0.997), and its online adaptivity enables it to track shifting workloads---making it the practical default for deployment.
We acknowledge that the core scheduling idea is shared with concurrent work (ContextPilot~\citep{contextpilot2025}); our contribution is the \emph{deployment-ready realization}: a self-contained, engine-agnostic implementation with formal approximation guarantees, a closed-loop feedback mechanism, and comprehensive cross-workload/cross-architecture evaluation demonstrating that this simple approach is sufficient for practical RAG serving.

\textbf{Relation to RAGCache.} Our knowledge tree is inspired by RAGCache's~\citep{jin2025ragcache} document-indexed trie structure, but serves a different purpose. RAGCache uses the tree for custom KV cache management within a modified serving engine; we use it solely for ordering decisions, relying on the engine's built-in APC for cache management. Our trie-based ordering achieves 97.5\% of oracle's TTFT reduction with $O(k)$ cost, confirming near-optimal scheduling quality. Our approach is complementary to RAGCache: it achieves comparable scheduling benefits without requiring engine modifications, and could be combined with RAGCache's cache management for further gains.

\textbf{Relation to CacheBlend.} CacheBlend~\citep{yao2025cacheblend} addresses a harder problem: reusing KV blocks regardless of position via selective recomputation. Our approach avoids this complexity entirely by converting the problem to pure prefix matching through ordering. The two approaches are complementary: CacheBlend could handle the non-prefix portion of our reordered sequences for further gains.

\textbf{Limitations.}
\begin{itemize}
\item \textbf{Controlled workloads}: Our main benchmark uses synthetic documents with controlled overlap. We supplement with a TF-IDF-retrieved diverse corpus (Section~\ref{sec:real_corpus}), a geospatial workload (Section~\ref{sec:spatial}), and NQ-Open traces (Section~\ref{sec:nq}), but production multi-tenant RAG traces remain future work.
\item \textbf{Concurrency scope}: Batched evaluation (Table~\ref{tab:concurrent}) shows ordering benefits at concurrency 1--8, and cache pressure sweeps confirm robustness across memory levels. However, we do not evaluate under sustained multi-tenant load with realistic arrival distributions (e.g., Poisson arrivals, heterogeneous query patterns), where trie staleness and scheduling interactions may differ.
\item \textbf{Throughput is decode-dominated}: Ordering improves prefill, not decode. Under workloads with long output sequences (e.g., summarization), the TTFT benefit shrinks as a fraction of total latency.
\item \textbf{Model family}: We evaluate Qwen2.5 at three scales (0.5B, 1.5B, 7B) and Microsoft Phi-3 (3.8B), observing consistent 18--33\% benefits across architectures. Validation on further model families (e.g., Llama, Mistral) remains future work.
\end{itemize}

\textbf{Future work.} Promising extensions include: (1)~combining ordering with CacheBlend-style selective recomputation for non-prefix documents, (2)~extending the TTFT-based feedback loop to query vLLM's block table directly for more precise cache-state tracking under sustained high-concurrency load, (3)~real-corpus evaluation on production RAG workloads with natural temporal locality, and (4)~multi-engine validation (SGLang RadixAttention).

\section{Conclusion}
\label{sec:conclusion}
% ================================================================

We presented RAGCache++, a prompt-layer scheduling policy that improves KV cache prefix reuse in RAG serving by reordering documents to extend cached prefixes. Under bursty document overlap, RAGCache++ reduces median TTFT by 20--33\% over retrieval-order APC across two GPU configurations (statistically significant: $16.8 \pm 1.6$\,ms, 95\% CI excludes zero), saving 66\% of prefill computation at p50 while adding $<$0.13\% overhead, zero throughput regression, and zero GPU memory cost. End-to-end pipeline measurements confirm that benefits persist when retrieval latency is included (16--26\% at $T_{\text{ret}} = 10$--$50$\,ms). The knowledge tree's predicted reuse correlates with observed reuse at $r = 0.97$, validating the approximate cache index design. The greedy ordering achieves 97.5\% of oracle's TTFT reduction (prefix-length optimality ratio 0.997), benefits persist under concurrent load (7--10\% at batch sizes 1--8) and are robust to cache pressure (29--31\% across GPU memory levels). On a geospatial workload with naturally lower overlap (Jaccard~$= 0.17$), ordering still provides measurable gains (4.5\% TTFT reduction). The approach requires no serving-engine modifications (${\sim}$150 LOC for the core ordering policy; ${\sim}$2{,}200 LOC for the full integrated system). Our results demonstrate that prefix \emph{depth}---not just hit count---is the key driver of latency reduction in prefix-caching systems.
